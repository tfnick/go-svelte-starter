# Marketing Landing Architecture Research

## Question

How should this Go + Svelte single-binary starter add SEO-capable marketing pages while preserving one-click deployment for a SaaS app?

## Repo Context

* Production embeds `frontend/dist` into the Go executable via `//go:embed frontend/dist` in `static.go`.
* `registerFrontendRoutes` currently serves every non-`/api/*` path from embedded Svelte assets and falls back to `frontend/dist/index.html`.
* The Svelte router maps `/` to the logged-in dashboard. This makes the public root page a SPA shell, not a semantic marketing document.
* `Makefile`, `build.bat`, `Dockerfile`, and `verify-build.bat` all assume `frontend/dist` is built before `go build`, then embedded into one binary.
* Existing tests assert `/login` and other SPA paths return the embedded Svelte `index.html`, and `/api/*` is not swallowed by the frontend fallback.
* Existing checkout APIs are protected: `POST /api/orders` creates a pending local order for a logged-in user, and `POST /api/orders/:id/payment-checkout` creates the Creem checkout.
* Products already carry checkout metadata such as `creem_product_id`, `billing_type`, `membership_level`, and `subscription_interval`.

## External Notes

* Google Search documentation recommends making important page content available in the rendered HTML for reliable indexing. Server-rendered or statically rendered HTML is safer for marketing pages than an empty SPA shell.
* htmx can be adopted progressively by adding its script and `hx-*` attributes to ordinary HTML, so it fits a server-rendered landing page without requiring a second frontend framework.
* Go `html/template` is a standard-library fit for trusted server-owned templates because it contextually escapes HTML, attributes, CSS, and URLs.
* Vite can build multiple HTML entry points, but using Vite for marketing pages keeps those pages inside the frontend toolchain and does not solve server-side metadata or runtime content updates by itself.

Useful references:

* https://developers.google.com/search/docs/crawling-indexing/javascript/javascript-seo-basics
* https://htmx.org/docs/
* https://pkg.go.dev/html/template
* https://vite.dev/guide/build#multi-page-app

## Feasible Approaches

### Approach A: Go-rendered marketing pages plus Svelte SaaS app

How it works:

* Add a `marketing/` or `web/marketing/` template directory embedded into Go.
* Serve `/`, `/pricing`, `/features`, `/blog/...`, `/robots.txt`, and `/sitemap.xml` before the Svelte SPA fallback.
* Move the SaaS dashboard root from `/` to `/app` or `/dashboard`; keep auth routes such as `/login` on the SPA.
* Use `html/template` for layout, metadata, Open Graph tags, canonical URLs, and structured data.
* Optionally add htmx on public pages for newsletter signup, waitlist, contact forms, pricing toggles, and lightweight partial updates.

Pros:

* Best SEO baseline because crawlers receive real HTML.
* Preserves one binary deployment; templates and assets can be embedded just like Svelte dist.
* Keeps SaaS SPA architecture unchanged behind an `/app` boundary.
* Allows runtime site settings to feed marketing metadata without rebuilding JS.

Cons:

* Introduces a second rendering style: server-rendered HTML for public marketing and Svelte for app.
* Requires route boundary migration because `/` currently means Dashboard in the Svelte router.
* Needs shared design tokens/assets discipline so marketing and SaaS do not visually diverge.

### Approach B: Vite multi-page static marketing plus Svelte SPA

How it works:

* Configure Vite to emit `index.html` for landing and another SPA entry such as `app.html`.
* Embed the output in Go and serve static marketing HTML at `/`.
* Keep most authoring in the frontend toolchain.

Pros:

* Fewer Go template concepts.
* Can reuse Tailwind build and frontend assets easily.
* Good for fully static landing pages.

Cons:

* Runtime metadata/content editing is harder.
* htmx/server partials become awkward because public pages are generated by Vite, not Go.
* Still requires careful route fallback split between marketing static files and app SPA.

### Approach C: Keep Svelte SPA and add prerender/SSR toolchain

How it works:

* Migrate to SvelteKit or add a prerendering/SSR build step for landing routes.
* Go serves generated HTML and app assets, or delegates to a Node server in deployment.

Pros:

* Single frontend mental model.
* Mature routing/metadata conventions if SvelteKit is adopted.

Cons:

* Larger migration and build/deployment complexity.
* A Node runtime would undermine the current one-binary deployment goal.
* Static prerender alone does not give htmx-style server interactions.

## Recommendation

Use Approach A for MVP: Go-rendered marketing pages with optional htmx, and move authenticated SaaS to `/app` or `/dashboard`.

This matches the product goal most directly: SEO-capable public pages, one executable, and an integrated SaaS backend. It also keeps the Svelte app focused on authenticated, interactive workflows where SPA ergonomics matter.

For checkout integration, the lower-risk MVP is a logged-in checkout continuation:

* Marketing CTA links to `/app/checkout?product_id=<local-product-id>` or `/login?redirect=/app/checkout?...`.
* The SPA preserves/reads the selected product after login, calls the existing protected `CreateOrder` and `CreateOrderPaymentCheckout` APIs, then redirects to Creem.
* This avoids anonymous order ownership, guest account creation, webhook reconciliation ambiguity, and public API abuse in the first cut.

Anonymous public checkout can be a later enhancement if the product requires guest purchases. It would need a new public route/usecase with idempotency, email capture, abuse controls, order ownership reconciliation, and clear post-payment account linking.

## Open Decisions

* Preferred SaaS app base path: `/app` or `/dashboard`. Decision: `/app`.
* Whether MVP should include only `/` landing page or also `/pricing`, `/features`, `robots.txt`, and `sitemap.xml`.
* Whether marketing content is code-defined templates only, DB/site-settings driven, or file/Markdown driven.
* Checkout integration depth: logged-in continuation for MVP, or anonymous public checkout in the first release.
